优化项目 1：detectHosts() 检查逻辑简化（第一版优化）
================================================================================

【位置】
HostManager.cpp 第 324-335 行

【原代码（复杂版本）】
```cpp
// 1. 遍历已保存的主机信息，检查是否有控制器实例存在
QList<uint8_t> hostWithInstance;       // 存储有实例的主机地址
for (auto it = m_hostInfoMap.begin(); it != m_hostInfoMap.end(); ++it) {
    const HostInfo& info = *it;

    if (m_controllers.contains(info.address)) {
        hostWithInstance.append(info.address);
        qDebug() << "HostManager: 检测到主机 0x"
                 << QString::number(info.address, 16).toUpper()
                 << "有控制器实例，状态：" << info.statusText();
    }
}

// 2. 检测结果如过存在实例，发出错误信息，返回空列表表示失败
if (!hostWithInstance.isEmpty()) {
    // 🔥 有任何主机实例存在，禁止探测
    qWarning() << "HostManager: 检测到以下主机有控制器实例:"
               << hostWithInstance
               << "，禁止执行探测操作";

    // 构建详细的错误信息
    QStringList hostDetails;
    for (uint8_t addr : hostWithInstance) {
        HostInfo* info = getHostInfoPtr(addr);
        if (info) {
            hostDetails << QString("0x%1 (%2)")
            .arg(addr, 2, 16, QChar('0')).toUpper()
                .arg(info->statusText());
        }
    }

    emit errorOccurred(QString("有主机控制器实例未释放，无法探测:\n%1\n\n"
                               "请先停用或移除所有主机后再探测。")
                           .arg(hostDetails.join("\n")));

    return QList<uint8_t>();  // 返回空列表表示失败
}
```

**代码行数**：37 行  
**时间复杂度**：O(n)，需要遍历 m_hostInfoMap  
**空间复杂度**：O(n)，需要临时列表 hostWithInstance  

---

【优化后代码（简洁版本）】
```cpp
// 👇 简化：直接检查是否有控制器实例存在
if (!m_controllers.isEmpty()) {
    qWarning() << "HostManager: 检测到有主机控制器实例未释放，禁止执行探测操作";
    emit errorOccurred("有主机控制器实例未释放，无法探测\n\n"
                       "请先停用或移除所有主机后再探测。");
    return QList<uint8_t>();  // 返回空列表表示失败
}
```

**代码行数**：5 行  
**时间复杂度**：O(1)，直接检查容器  
**空间复杂度**：O(1)，无需临时变量

【优化效果对比】

| 指标 | 原代码 | 优化后 | 改进 |
|------|--------|--------|------|
| 代码行数 | 37 行 | 5 行 | **减少 32 行 (86%)** |
| 时间复杂度 | O(n) | O(1) | **性能提升** |
| 空间复杂度 | O(n) | O(1) | **内存节省** |
| 临时变量 | 2 个 | 0 个 | **更简洁** |
| 可读性 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | **显著提升** |
| 维护成本 | 高 | 低 | **易于维护** |

【优化理由】

1. **语义等价性** ✅
   - 原逻辑：检查是否有任何控制器实例存在
   - 新逻辑：检查 m_controllers 是否为空
   - 两者完全等价，因为 m_controllers 存储所有实例

2. **业务逻辑正确性** ✅
   - 探测前必须确保没有遗留的控制器实例
   - **任何**实例存在都应该阻止探测
   - 不需要知道具体是哪个地址的实例（后续会全部清空）

3. **与后续代码一致** ✅
   - 第 370-387 行会清空所有数据：
     ```cpp
     m_hostInfoMap.clear();
     for (QObject* controller : m_controllers) {
         controller->deleteLater();
     }
     m_controllers.clear();
     ```
   - 所以前面的详细检查是冗余的

4. **用户建议的核心思想** 💡
   > "这一步判断是否存在主机控制器实例，是不是直接检查 m_controllers 是否为空会更简单？"
   > "因为任意实例存在都应该先释放，然后才能探测？"
   
   这个建议非常准确！抓住了问题的本质。

优化项目 2：删除多余的检查逻辑（第二版优化）
================================================================================

【问题发现】
用户深入思考后提出关键问题：
> "这里需要仔细考虑，一般来说有两种情况：
> 1. 第一次探测，不存在任何的主机信息，这时候也不存在任何主机控制器，可以开始探测保存探测结果；
> 2. 程序在运行过程中执行探测，这时候可能存在已经探测的地址，甚至控制器，应该是在存在控制器的时候不能执行探测。
> 是不是确实只要判断 m_controllers 既可？"

【深度分析】

场景 1：冷启动探测（首次探测）
- m_hostInfoMap: 空
- m_controllers: 空
- 检查结果：m_controllers.isEmpty() → true ✅
- 结果：通过检查，开始探测 ✅
- 评价：正常工作

场景 2：热重启探测（运行中重新探测）
- m_hostInfoMap: {0x01, 0x02}
- m_controllers: {0x01: XdcController*} ← 有实例！
- 检查结果：m_controllers.isEmpty() → false ❌
- 结果：阻止探测，报错退出 ❌
- 评价：**无法重新探测！这是 Bug！**

【关键发现】
查看 detectHosts() 第 337-355 行的清理逻辑：
```cpp
// 清空之前的响应记录和主机信息
{
    QMutexLocker locker(&m_detectHostMutex);
    m_detectHostResponse.clear();
    
    // ⭐ 清除所有主机信息
    m_hostInfoMap.clear();  // ← 自动清空旧数据
    
    // ⭐ 销毁并清除所有控制器实例
    for (QObject* controller : m_controllers) {
        controller->deleteLater();
    }
    m_controllers.clear();  // ← 自动销毁旧实例
    
    m_activeHostAddress = 0;
}
```

**结论**：
- ✅ 探测函数本身会**自动清理**所有旧数据和实例
- ❌ 提前检查是**多余且有害**的
- ❌ 会导致无法在运行中重新探测

【原代码（第一版优化后）】
```cpp
// 👇 简化：直接检查是否有控制器实例存在
if (!m_controllers.isEmpty()) {
    qWarning() << "HostManager: 检测到有主机控制器实例未释放，禁止执行探测操作";
    emit errorOccurred("有主机控制器实例未释放，无法探测\n\n"
                       "请先停用或移除所有主机后再探测。");
    return QList<uint8_t>();
}
```

问题：
❌ 阻止热重启探测
❌ 与后续清理逻辑矛盾
❌ 用户体验差（必须先手动移除所有主机）

【优化后代码（第二版）】
```cpp
QList<uint8_t> HostManager::detectHosts(uint8_t maxAddress)
{
    // 保存旧的活动主机地址（用于信号发射）
    uint8_t oldActiveAddress = m_activeHostAddress;

    // 清空之前的响应记录和主机信息
    {
        QMutexLocker locker(&m_detectHostMutex);
        m_detectHostResponse.clear();

        // ⭐ 清除所有主机信息（无论是否有实例）
        m_hostInfoMap.clear();

        // ⭐ 同时销毁并清除所有控制器实例
        for (QObject* controller : m_controllers) {
            if (controller) {
                controller->deleteLater();
            }
        }
        m_controllers.clear();

        // ⭐ 清空活动主机地址
        m_activeHostAddress = 0;
    }

    // 如果之前有活动主机，发射信号通知 QML
    if (oldActiveAddress != 0) {
        emit activeHostChanged(oldActiveAddress, 0);
    }

    qDebug() << "HostManager: 开始探测主机，地址范围：0x01 -" << QString::number(maxAddress, 16).toUpper();
    // ... 后续探测逻辑
}
```

优势：
✅ 支持冷启动探测
✅ 支持热重启探测
✅ 自动清理，用户友好
✅ 符合"探测即重置"的语义
✅ 代码更简洁（删除 8 行冗余检查）

【行为对比】

| 场景 | 原代码（第一版优化） | 新代码（第二版优化） |
|------|---------------------|---------------------|
| 冷启动探测 | ✅ 正常 | ✅ 正常 |
| 热重启探测 | ❌ 阻止，报错 | ✅ 自动清理后重新探测 |
| 用户体验 | ⭐⭐ (需手动移除) | ⭐⭐⭐⭐⭐ (一键重扫) |
| 代码行数 | 减少 32 行 | 再减少 8 行 |

【设计哲学】

探测函数的正确语义应该是：
> **"重新扫描网络上的所有主机，替换旧的扫描结果"**

这意味着：
- ✅ 不关心之前有什么数据
- ✅ 无条件清空旧数据
- ✅ 执行全新的扫描
- ✅ 保存新的扫描结果

类比手机"恢复出厂设置"：
- ❌ 错误做法：检查是否有应用，有就阻止重置
- ✅ 正确做法：直接清空所有数据，重新开始
